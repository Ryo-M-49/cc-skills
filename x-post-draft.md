# Xポスト下書き生成

`devstories/x-buzz-research/analysis.md` のバズパターン分析に基づき、devstoriesアカウント用のバズ狙い投稿を下書きする。

## 実行フロー

### Step 1: ネタの確認

ユーザーから投稿のネタ（海外事例、ツールリリース、気づき等）を受け取る。
ネタがない場合は、最近のdevstories記事やリサーチ結果からネタを提案する。

### Step 2: 型の選択

ネタに最も合う型を選び、ユーザーに提案する。

| 型 | 使うとき | 平均rate |
|---|---|---|
| ① リリース爆弾型 | プロダクトを出した/更新した時 | 1.39x |
| ② 知ってた？型 | 海外事例やツールを紹介する時 | 0.34x |
| ③ 一行断言型 | 洞察や意見を述べる時 | 0.40x |
| ④ あるある共感型 | 開発者の日常ネタの時 | 0.91x |
| ⑤ 仕組み公開型 | 自分のワークフローを見せる時 | 0.30x |
| ⑥ ネタ・エンタメ型 | 面白いアイデアがある時 | 1.0x+ |

### Step 3: 下書き生成

選んだ型のテンプレートに沿って**3案**書く。おすすめ1案を明示。

### フック設計（最重要）

1ポスト目の最初の1〜3行（15文字〜140字）でスクロールを止める。ここが全体のインプレッションを決める。

**フックの型（効果が高い順）**
- **数字・具体型**: 「売上$401M。従業員2人。初期投資$20K。」「$29→$79に値上げ。ユーザー40%減。でも売上35%増。」
- **逆説・本音突き型**: 「"Build in public"がスタートアップを殺しかけた」「一番リスクが高いのは技術者だ」
- **問いかけ型**: 「$23K MRR、週25時間。これ以上スケールする意味ある？」
- **共感・痛み型**: 「共同創業者が40%の株を持って消えた。ベスティング契約はなかった」
- **Before-After型**: 「$200の見積もりを出そうとした。結果、$15Kで即決された」

**フック設計のルール**
- 最初の1行は15文字前後で勝負。読者は一瞬で「読む/読まない」を判断する
- 抽象語で始めない。数字・固有名詞・具体的な状況から入る
- 140字付近で「さらに表示」が出るように設計する。切れ目が気になって開かせる
- 感情の引き金を引く: 驚き・共感・好奇心のどれかを最初の3行で起動させる
- 改行を活用して1文を短く区切る。テンポの良さがスクロール停止率を上げる

**NGフック**
- 抽象的すぎる（「AIの時代が来ている」→ 誰でも言える）
- 宣伝臭が強い（「新サービスリリース！」→ スルーされる）
- 続きが想像できる（「SEOが大事という話」→ 読む動機がない）
- 対象が広すぎる（「すべてのビジネスマンへ」→ 誰にも刺さらない）

**フック→本文→CTAの連動**
- フック（止める）→ 価値提供（滞在させる）→ CTA（行動させる）
- フックだけ強くて中身が薄いと逆効果。滞在時間が短いとアルゴリズムに嫌われる
- 投稿後30分〜2時間の初速が重要。リプライを誘う問いかけやCTAで初速を作る

### 黄金フォーマット（全型共通の制約）

- **文字数**: 100-250字（中文が最もバズりやすい）
- **行数**: 3-7行
- **メディア**: ありが望ましい（rate 0.291 vs 0.111）
- **1行目**: 上記フック設計に従う。数字・具体性・感情の引き金を最優先
- **1ツイート目で「一瞬で価値が伝わる」ようにする**: 長文スレッドでも最初の数行で結論＋面白ポイントを出す。タイムラインをスクロールしてる人は最初の数行で読む価値を判断する。衝撃の事実・具体的な数字・キャッチーな一文から始めて、技術解説はスレッドの2〜3ツイート目に回す
- **数字を入れる**: 金額、時間、人数など具体的な数字
- **CTA**: 最終行に一言（質問、感想、リンク）

### 各型のテンプレート

**① リリース爆弾型**
```
【リリースしました】 or 〇〇を作りました！
[何ができるか1行]
[面白い体験・差別化を1行]
ぜひ使ってみてください！
[URL]
#個人開発 [スクショ or デモ動画]
```

**② 知ってた？型**
```
[驚きの事実]って知ってた？ or [数字]がヤバい。

[具体的に何が起きてるか 2-3行]

[自分の意見 or やり方の入口]
```

**③ 一行断言型**
```
[30-80文字で断言。メディアなし。議論を呼ぶが炎上しないライン]
```

**④ あるある共感型**
```
[個人開発あるある or エンジニアあるある]、[具体的な状況]
or
[質問形式: 「みんなどうしてる？」]
```

**⑤ 仕組み公開型**
```
[結論: 〇〇を自動化した / 〇〇のルーティン公開]

・項目1
・項目2
・項目3

[感想 or 結果の数字]
```

## トーン・文体ルール（最重要）

### 目標トーン
エンジニア・個人開発者・インディー開発者向け。クールでストレート、親しみやすいタメ口。謙虚すぎず、押し売りせず、実用的で「自分も再現できそう」と思わせる。情報密度が高く、読み飛ばしにくい。

### 文体統一（最重要）
- **完全に断定調・タメ口で書く。ですます調は一切使わない**
- 「〜です」「〜ます」「〜と思います」「〜らしい」を徹底的に排除
- 例：「これは〜だ」「必要だ」「成果が出るまで〜かかる」など、ストレートに書く
- 曖昧語（思う、らしい、など）を減らし、断定的に書く

### タイトルと冒頭の強化
- タイトル：数字の衝撃（$25K、1人、100%インバウンドなど）を入れ、具体的にする
- 冒頭：最初の2〜3行で最大のフック（数字＋意外性）を入れ、読者を引き込む
- 海外事例は「誰が・何を・どれくらい達成したか」を早めに提示

### 読みやすさの構造化
- 適度な改行（1文が長くなりすぎない）
- 長い段落は分割して「テキストの壁」を避ける
- **Xスレッド（daily投稿）ではMarkdown記法は使えない**。太字・見出し・箇条書き記号は無効。改行と「・」で構造を作る
- note記事やWeekly投稿など、Markdownが使える媒体では太字・見出し・箇条書きを積極的に活用する

### 内容のブラッシュアップ
- 冗長な表現・重複を削る（情報密度を上げる）
- 抽象的な表現は具体例や数字で裏付ける
- 「再現するなら」の部分はチェックリスト形式で実践しやすく
- 読者が「自分ごと化」できるように「日本でやるなら」「再現するなら」のセクションを入れる
- コスト感や数字の内訳は箇条書き・太字で読みやすく表記
- 全体の長さはX長文投稿として読みきれる程度に（必要なら自然に削る）

### devstories固有ルール
- 海外事例や海外の定説を1つ入れる（差別化ポイント）
- 上から目線にならない。「自分もやってみた」のスタンス
- 宣伝臭を出さない
- 専門用語は使ってOK（ターゲットがエンジニア/開発者）

### devstories誘導文（最終ポストに使用）

前半1文（スレッド連動）＋ 後半1文（固定）のハイブリッド構成。タメ口で統一し、宣伝感を消す。

構成:
```
[前半: そのスレッドの内容に自然に繋がる1文。「〜の深掘りはdevstoriesでやってる」「〜みたいな事例を毎日紹介してる」など]
[後半（固定）: 海外で個人が静かに稼いでるビジネスを毎日紹介中 →https://devstories.me]
```

例:
```
TinyWowみたいな無料ツール戦略の深掘りはdevstoriesでやってる。海外で個人が静かに稼いでるビジネスを毎日紹介中 →https://devstories.me
```

NG:
- ですます調（スレッド本文と合わなくなる）
- 毎回同じ定型文のコピペ（フォロワーが2回目以降スルーする）

### シリーズ統一ルール
- 出典は最後に明確にまとめる
- シリーズ感を出す場合は「Vol.002:」などを残しつつ魅力的に

### 出力先

`devstories/drafts/YYYY-MM-DD/x-post.md` に保存。

## オプション引数

- `--type リリース`: 型を指定（リリース/知ってた/断言/あるある/仕組み/ネタ）
- `--topic "海外事例名"`: ネタを指定
- `--count 5`: 案の数を変更（デフォルト3）

## 重要

- human-writingスキルを適用すること（AI臭を消す）
- 3案のうち1案は必ず「一行断言型」で書く（最も手軽にテストできるため）
